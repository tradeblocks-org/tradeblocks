# TradeBlocks MCP Server 3.0 — Market Data 3.0 Infrastructure

Market data can now live as Parquet files instead of DuckDB rows — readable by any tool that speaks Parquet (DuckDB, Polars, pandas, Arrow), inspectable as plain folders, and free from the single-writer lock that DuckDB enforces. ThetaData is a first-class provider alongside Massive, with wildcard-bulk streaming — for quotes and now daily open interest — that turns what used to be thousands of API calls into one. And `import_flat_file` was rewritten so you can point it at any Parquet or CSV file and Claude figures out how to ingest it — no provider-specific paths required.

This is **3.0**. The market-data schema and tool surface are stable; the sections below trace what landed across the betas on the way here, newest first.

---

# What's New Since Beta 5

The changes that close out the road to 3.0: daily option open interest joins the market datasets, the greeks vega unit is pinned and documented, and a round of ingest/read-path hardening.

## Daily option open interest

Open interest is now a first-class market dataset. `market.option_oi_daily` (Hive-partitioned by underlying and date) stores end-of-day open interest per contract, fetched from ThetaData with the same single-wildcard-stream-per-root pattern that powers bulk quotes — one request returns every strike's OI for an underlying on a given date. It writes alongside `option_chain` and `option_quote_minutes` through the same store abstraction, so anything that already reads chains or quotes can join open interest by contract without a new access path.

## Vega is per-1% IV move — pinned and documented

`option_quote_minutes` stores vega normalized to a **1% change in implied volatility** (not a full 1.00 move), and the column metadata now states this explicitly. The value is uniform across every partition, so downstream consumers read vega directly without guessing the unit or rescaling — no per-row normalization needed.

## Ingest and read-path hardening

A round of robustness fixes for long historical backfills and multi-hour read workloads:

- **Market ingestor + ThetaData hardening.** A batch of fixes to the ingest path and the ThetaData provider that remove failure modes hit during real backfills, so a long ingest is less likely to abort partway.
- **Anomalous-quote detection.** A shared `isAnomalousQuote` check flags blown or crossed quotes (`bid > ask`, zero/garbage spreads), and the trade-replay path now guards its mark against blown spreads — falling back to an `HL2` midpoint when the tape is degenerate instead of trusting a bad print.
- **Read-path resilience, continued.** The hot quote-read paths were further hardened against the long-session DuckDB driver flake first addressed in Beta 3, inlining typed literals on the remaining read queries so multi-hundred-call workloads run clean.

# Analysis & Replay Tools

3.0 also changes how the trade-analysis and replay tools — `replay_trade`, `analyze_exit_triggers`, `batch_exit_analysis`, `decompose_greeks` — source their data and evaluate exit rules.

## Replay and exit tools read local data only

These tools now price exclusively from your **pre-ingested local market data** — option legs from stored mid quotes, underlying and VIX bars from the stored daily/minute datasets (with a daily-aggregate fallback). The previous fetch-on-cache-miss path is gone. Backfill the dates a trade spans with the market-data tools first; on missing data these tools return empty or degraded results rather than fetching in the background. Two related touch-ups: `replay_trade`'s `skip_quotes` argument is now a no-op for option legs (still accepted, ignored), and underlying bars with zero or non-finite prices are filtered before greeks, so bad spot data can't produce nonsense greeks.

## Exit-trigger refinements

Two changes to how candidate exit rules evaluate against a replayed P&L path:

- **`slRatioMove`** now triggers on a **directional percentage move** (sign-aware) rather than an absolute distance — the threshold reads relative to the entry ratio, so exits fire at more intuitive points.
- **`perLegDelta`** now sign-adjusts per-leg delta for short legs (negated when the leg is sold) before testing the threshold, so delta-based exits fire correctly on short legs.

Mark pricing across these tools also gained the crossed/blown-quote `HL2` guard noted above, so a degenerate exchange print no longer distorts the replayed path.

# What's New in Beta 5

`fetch_quotes` now works on Massive Developer plans. Previously it returned `status: "unsupported"` whenever `MASSIVE_DATA_TIER` wasn't set to `quotes`, because the dispatch gate was checking "is real NBBO available?" instead of "can we fetch quote data at all?". On lower tiers `fetch_quotes` now falls back to `/v2/aggs` minute bars and synthesizes `bid=ask=close` per minute — downstream greeks fitting averages bid+ask, so close serves as the mid proxy.

To keep the two cases distinguishable, `option_quote_minutes` now populates the `source` column for Massive rows: `'nbbo'` for true bid/ask via `/v3/quotes`, `'synth_close'` for the synthesized fallback. Filter by `source = 'nbbo'` if your analysis requires real spreads. Set `MASSIVE_DATA_TIER=quotes` to keep using true NBBO. ThetaData rows still write `NULL` to `source` pending a follow-up — its data is true NBBO regardless.

ThetaData history calls also now work against newer ThetaTerminal builds. Recent Terminal versions return JSON history responses in columnar shape (`{open: [...], close: [...], high: [...]}`) instead of the row-array shape the parser was built against, which surfaced as `ThetaData /v3/index/history/eod returned unexpected JSON shape` errors on every history call. The parser now detects and transposes the columnar variant before the downstream mappers see it; older shapes continue to work unchanged.

# What's New in Beta 3

Beta 3 focuses on the read path. Long-running quote reads no longer hit the DuckDB driver flake that aborted multi-hundred-call workloads, the chain-existence probe is now a single syscall instead of a SQL round-trip, and the default memory limit auto-scales to your machine instead of capping at 512MB. Option greeks are now computed and written inline on `market.option_quote_minutes`. RSI now uses the Wilder calculation, and `refresh_market_data` no longer wastes round-trips on weekends and holidays.

## Inline greeks on `market.option_quote_minutes`

The quote ingestor now computes Black-Scholes greeks (delta, gamma, theta, vega, rho) from each written quote row's midpoint and persists them inline. Downstream consumers no longer have to re-derive greeks from raw quotes — read the column. Partitions written before this change still work; missing values are filled in memory at query time.

## DuckDB read-path resilience

Three changes that together make multi-hour read workloads stop falling over:

- **Auto-scaled memory limit.** `DUCKDB_MEMORY_LIMIT` now defaults to 75% of total system RAM (DuckDB's own native default is 80%; we keep a small headroom for Node's heap and the OS). Floored at 1GB so small VMs and CI containers still work; no upper cap. The old hardcoded "512MB" default caused mid-run OOMs on real workloads.
- **`extract_statements` GC leak fixed.** Long read sessions used to die mid-run with `Failed to execute prepared statement`. Root cause: every parameterized `runAndReadAll(sql, values)` call goes through `node_bindings.extract_statements`, which returns a C++ handle with no destroy method on its JS wrapper — handles only release on JS GC. With unique partition paths embedded in each call, DuckDB caches a separate plan per call and the handles outpace GC after a few hundred calls. The hot read paths now use unbound `runAndReadAll(sql)` with inlined SQL literals (typed config, no injection vector), bypassing `extract_statements` entirely.
- **Filesystem chain probe.** `ParquetChainStore.hasChain(underlying, date)` now checks for partition existence with a single `existsSync` syscall instead of a SQL view query — drops from ~430ms to ~1ms per call, which adds up fast on tools that probe many dates.

## RSI calculation fix

`RSI_14` was using a simple moving average on gains and losses, which is wrong — the standard everyone else uses is Wilder smoothing. Switched. Existing partitions need re-enrichment to pick up the new values.

## `refresh_market_data` skips non-trading days

The composite daily-refresh tool now filters out weekends and US market holidays before fanning out to the provider. No more wasted round-trips for dates the exchange wasn't open.

## Data pipeline bug fixes

Two narrowly-scoped data-pipeline bugs found by [@goodorigamiman](https://github.com/goodorigamiman) (PR #278) — both silent (no errors, just bad/empty data):

- **`enrich_market_data` writes empty Parquet on first-time enrichment.** In Parquet mode the daily working table is seeded from existing `enriched/ticker=*/data.parquet` files. For a ticker whose seed contained zero rows, the subsequent UPDATE matched nothing and silently no-op'd, so the flush wrote a header-only Parquet (~675 bytes), perpetuating the empty state across every later enrichment call. Switched to `INSERT OR REPLACE` (with a UNIQUE INDEX on `(ticker, date)` on the working table to support it) — first-time enrichment now populates correctly.
- **VIX1D classified as stock instead of index.** `MarketIngestor.detectAssetClass` routed VIX1D fetches to `/v3/stock/history/eod`, which returns shells with all-zero OHLCV for unknown stock symbols. Every VIX1D row in `market.spot_daily` had `open=high=low=close=0`. Added VIX1D to the index ticker set; refetching the full history now writes real OHLCV.

# What's New in Beta 2

Beta 2 is about making the bulk fetch path robust for real historical backfills. Progress notifications turn multi-hour ingests from "is it hung?" into a visible progress bar. A ThetaData server-side bug that used to abort isolated historical dates now auto-routes around itself. A quote-filter fix recovers rows that were silently dropped near expiry. And two smaller improvements — bulk read primitives on the stores, and an opening/closing commission split on strategy profiles — land alongside.

## Progress notifications on long-running fetch_quotes

Bulk-underlying `fetch_quotes` calls for a single SPX day stream ~12M rows and take 1-3 minutes; a month of backfill runs for the better part of an hour. Previously the MCP client had no signal that anything was happening — the tool either returned a summary or timed out. Now `fetch_quotes` emits MCP `notifications/progress` events when the caller provides a `progressToken`, with determinate totals computed from provider capabilities (so progress bars render correctly without peeking at internals):

- One event per (root, right) wildcard group as it finishes — for SPX that's four events per date (SPX call, SPX put, SPXW call, SPXW put), in whatever order the concurrent streams complete.
- One event per date as its Parquet partition is flushed to disk.
- Per-ticker mode and calls without a `progressToken` silently skip the reporter — no overhead for the non-streaming case.

Claude Desktop and any MCP-aware client now show determinate progress for historical backfills. If you're writing custom MCP tooling, pass a `progressToken` in the tool call metadata to opt in.

## ThetaData resilience: per-expiration fallback when wildcard 500s

ThetaTerminal throws a server-side `NullPointerException` (surfaced as HTTP 500) on `/v3/option/history/quote?expiration=*&strike=*` for a small subset of historical dates — confirmed on 2022-07-11 and 2022-07-14 as of this release, and presumably others we haven't hit yet. The per-contract path works fine on the same dates, so the bug is isolated to the terminal's wildcard-expansion code.

Beta 2 adds an in-producer fallback to `fetchBulkQuotes`: on HTTP 500 from the wildcard stream, the provider enumerates expirations via `/v3/option/list/contracts/quote` (which works on the broken dates) and streams each expiration individually with `strike=*` still wildcarded. You get the same output and writer semantics; wall-clock for a broken date goes from "abort" to ~2-3× the normal bulk time (still bulk-over-strikes, just more round-trips). Happy-path dates are untouched — the fallback fires only on explicit 500s.

This was the missing piece that made long-horizon historical backfills tractable. Previously, any abort on one date would break a multi-month ingest; now the worst case is a few extra minutes on each affected date.

## Keep bid=0 quotes when ask > 0

Three places in the quote hydration path used to drop every row where `bid <= 0 || ask <= 0`, on the reasoning "one-sided NBBO would distort mid." That's wrong for deep-OTM options near expiry — `bid=$0.00, ask=$0.05` is the real tape (no bidders, but the option is still quoted to close), and the option has a legitimate mark. For near-ATM strikes that went bid-less for even one minute, this silently dropped 390+ rows per day on expiry day for every affected strike.

Beta 2 changes the filter to `ask <= 0 || bid < 0` — keep rows where `ask > 0`, drop only when there's genuinely no market. Going forward, new partitions preserve these rows. Existing partitions written before the fix remain swiss-cheese for affected strikes; full re-ingest is the only way to recover them.

## Bulk read primitives on the stores

The store abstraction gained three new methods for batched reads:

- `QuoteStore.readQuotesBulk(tickersByDate)` — single call returns quotes for many `(ticker, date)` pairs across many underlyings, with backend-native SQL (one `IN (...)` query per underlying rather than `N` per-date calls).
- `ChainStore.readChainBulk(underlying, dates)` — single-query bulk read for many dates.
- `SpotStore.readBarsBulk(tickers, from, to)` — same pattern for OHLCV bars.

Plus `groupTickersByUnderlying`, a shared helper that multi-date callers need before fanning out to backend-specific bulk paths (both `readQuotes` and `writeQuotes` enforce single-underlying batches).

Each store's abstract class provides a correct per-date fallback implementation, and the Parquet and DuckDB backends override with native bulk SQL. Opt-in perf logging via `MARKET_PERF_DEBUG=1`. No new MCP tool consumes these yet — they exist for downstream tools that need to read many partitions efficiently without repeating the per-date call pattern.

## Commission split on strategy profiles

`StrategyDefinition` and the profiles table now model opening and closing commissions separately, matching how Option Omega (and most brokers with asymmetric fees) bill:

- `opening_commission` and `closing_commission` are added as optional fields on the Zod schema, alongside the existing `commission_per_contract` default — backward-compatible for any caller still using the single-value field.
- The profiles DuckDB table gets the two new columns via `ALTER TABLE ADD COLUMN IF NOT EXISTS` — additive migration, no data loss on existing profiles.

If your broker charges the same on open and close, keep using `commission_per_contract` and nothing changes. If they charge asymmetrically (e.g., exchange fees only on the closing side), the new fields let you model it accurately.

# What's New in Beta 1

## Market Data 3.0: New Canonical Table Names

The market data layer has been reorganized around five canonical datasets, each with a single source of truth:

- `market.spot` — minute OHLCV bars, ticker-partitioned
- `market.spot_daily` — RTH-aggregated daily OHLCV, derived from `market.spot`
- `market.enriched` — per-ticker computed indicators (RSI, ATR, IVR/IVP, etc.)
- `market.enriched_context` — cross-ticker regime fields (Vol_Regime, Term_Structure_State, VIX ratios)
- `market.option_chain` / `market.option_quote_minutes` — chain metadata and NBBO minute quotes

Here's how they relate:

```
  PROVIDER FETCH                         DERIVED LOCALLY
  ──────────────                         ───────────────

  market.spot ──────────────────────►  market.spot_daily
  (minute OHLCV,                        (RTH daily aggregation,
   per ticker + date)                    view over spot)

         │
         └────────────────────────────►  market.enriched
                                          (per-ticker indicators:
                                           RSI, ATR, IVR/IVP,
                                           realized vol, gap, etc.)

                                                │
                                                ▼
                                         market.enriched_context
                                          (cross-ticker regime fields;
                                           reads VIX / VIX9D / VIX3M
                                           from market.enriched)

  market.option_chain                    (leaf — contract metadata)
  (per underlying + date)

  market.option_quote_minutes            (leaf — NBBO minute quotes)
  (per underlying + date)
```

The legacy names `market.daily`, `market.intraday`, and `market.date_context` are retired. The server drops them on open if they exist from a 2.x database. Migrate queries to `market.spot_daily`, `market.spot`, and `market.enriched_context` respectively.

## Parquet-First Storage (Optional)

Set `TRADEBLOCKS_PARQUET=true` to write market data as Parquet files instead of DuckDB rows. The directory layout is Hive-partitioned, so folders are self-describing and tools that speak Parquet can read them directly. Here's what the data root looks like with a couple of tickers ingested:

```
~/tradeblocks-data/
│
├── database/                            # Local DuckDB files (keep local per machine)
│   ├── analytics.duckdb
│   └── market.duckdb                    # Only written to when TRADEBLOCKS_PARQUET != "true"
│
├── market/                              # Parquet market data (portable / inspectable)
│   │
│   ├── underlyings.json                 # Ticker registry (OCC root → canonical underlying)
│   │
│   ├── spot/                            # → market.spot  (minute OHLCV)
│   │   ├── ticker=SPX/
│   │   │   ├── date=2026-04-14/data.parquet
│   │   │   └── date=2026-04-15/data.parquet
│   │   └── ticker=VIX/
│   │       └── date=2026-04-15/data.parquet
│   │
│   ├── enriched/                        # → market.enriched  (per-ticker indicators)
│   │   ├── ticker=SPX/data.parquet
│   │   ├── ticker=VIX/data.parquet
│   │   └── context/                     # → market.enriched_context  (cross-ticker regime)
│   │       └── data.parquet
│   │
│   ├── option_chain/                    # → market.option_chain  (contract metadata)
│   │   └── underlying=SPX/
│   │       ├── date=2026-04-14/data.parquet
│   │       └── date=2026-04-15/data.parquet
│   │
│   └── option_quote_minutes/            # → market.option_quote_minutes  (NBBO quotes)
│       └── underlying=SPX/
│           ├── date=2026-04-14/data.parquet
│           └── date=2026-04-15/data.parquet
│
├── market-meta/                         # Operational metadata (JSON, atomic writes)
│   ├── sync-metadata.json               # Per-source sync state (watermarks, content hashes)
│   ├── flat-import-log.json             # Dedupe ledger for import_flat_file calls
│   └── enrichment-watermarks.json       # Per-ticker last-enriched date
│
└── blocks/                              # Trade data and strategy profiles
    └── <block_id>/
        ├── trade-log.csv
        ├── profiles/<strategy>.json     # Strategy profiles (migrated from DuckDB in 3.0)
        └── .sync-meta.json              # Per-block sync state
```

A few things that get easier as a result:

- **Inspectable on disk.** Want to know what you have cached? `ls` the folder. Each partition is its own file; no opaque database blob.
- **No single-writer lock.** DuckDB enforces exclusive access to its files, which makes concurrent readers awkward. Parquet doesn't — multiple processes can read the same partition file at once.
- **Portable if you want it.** If you want the bars on another machine, copy `market/`. If you don't, the files just sit there.

Reads auto-detect: on startup, the server scans for `spot/ticker=*/` directories and registers views when it finds them. No env flag needed on the read side. Writes use `TRADEBLOCKS_PARQUET=true`; without the flag, writes go to physical DuckDB tables in `database/market.duckdb` (the 2.x behavior, still supported).

## New Market Ingestion Tools

Six new MCP tools handle fetching and refreshing market data:

- `fetch_bars { tickers, from, to, timespan }` — OHLCV bars for any supported ticker at daily or intraday granularity. Auto-runs per-ticker enrichment after writing (RSI, ATR, EMA, realized vol). Set `skip_enrichment: true` to defer.
- `fetch_quotes { tickers | underlyings, from, to }` — NBBO minute quotes. Per-ticker mode works on any provider; underlying mode uses wildcard bulk endpoints (ThetaData) when supported and returns `unsupported` on per-ticker-only providers (Massive).
- `fetch_chain { underlyings, from, to }` — option contract metadata (strike, expiration, type) for each trading date in the range.
- `compute_vix_context { from, to }` — cross-ticker regime fields (Vol_Regime, Term_Structure_State, Trend_Direction, VIX_Gap_Pct, VIX_Spike_Pct). Reads already-enriched VIX/VIX9D/VIX3M data — no provider call.
- `refresh_market_data { spot_tickers, chain_underlyings, quote_tickers, as_of }` — composite daily refresh. Fires `fetch_bars` for each spot ticker, `fetch_chain` for each underlying, `fetch_quotes` for each quote ticker, and auto-calls `compute_vix_context` when any VIX-family ticker is in the spot list. Returns a coverage report per ticker.
- `register_underlying` / `list_underlyings` — manage the OCC root → canonical underlying map (see below).

All of these route through the configured `MARKET_DATA_PROVIDER` (default `massive`; override per-call via the `provider` arg or globally via the env var).

## ThetaData Is a Full Provider

In 2.3, ThetaData was a stub — the flag existed but most calls returned empty. In 3.0, it's a full implementation you can switch to by setting `MARKET_DATA_PROVIDER=thetadata`:

- **OHLCV bars** (daily + intraday), with automatic monthly chunking and retry on HTTP 429 with exponential backoff.
- **Wildcard bulk quotes** — ThetaData's structural advantage. One request returns every strike's minute quotes for an underlying on a given date. A full SPX day that used to take thousands of per-contract API calls is now a single streaming request.
- **Option snapshots with greeks** (delta/gamma/theta/vega) from ThetaData's quote endpoints.
- **Historical contract chains** with `expired=true` support for historical strike lookups.
- **Auto-start Terminal.** The Java-based ThetaData Terminal is required for data access; the provider spawns and health-checks it automatically on first use, detects the 478 "multiple terminals" error, and handles graceful reconnection.
- **472 NO_DATA handling.** When ThetaData returns 472 (no data for this contract/date), the provider treats it as empty instead of erroring — matching Massive's semantics.

You can run both providers in the same TradeBlocks install and switch between them per-call via the `provider` arg on any fetch tool.

## Massive Gets Flat-File Imports

The Massive provider gained three new capabilities in 3.0:

- **Bulk flat-file downloads.** For a full day of option or index bars, the provider can pull the daily CSV.gz from Massive's S3 flat-files, filter to requested tickers, convert timestamps to ET, and write a Parquet file. A full SPX day finishes in ~30 seconds vs. thousands of per-ticker API calls.
- **Historical contract lists.** `fetch_chain` now works against Massive via the `/v3/reference/options/contracts` endpoint with `expired=true`, auto-paginating through historical chains.
- **Data availability awareness.** Massive declares 2023-02-14 as the earliest date for option flat-files — requests before that short-circuit as skipped instead of hitting S3 and failing.

The `MASSIVE_DATA_TIER` env var (`ohlc` | `trades` | `quotes`) from 2.3 continues to govern which API endpoints are available.

## Flat-File Imports Accept Any File

`import_flat_file` was rewritten around a simple premise: Claude has `describe_database`, `run_sql` with path-gated `read_parquet`/`read_csv`, and knows the store schemas. It can sniff any file, figure out which store it belongs to, and compose a SELECT that maps the file's columns to the target shape. The tool just dispatches.

The new signature:

```jsonc
{
  "file_path": "/path/to/any/file.parquet",
  "dataset_type": "spot_bars", // or "option_quotes" | "option_chain"
  "select_sql": "SELECT ticker, date, time, open, high, low, close, NULL::DOUBLE AS bid, NULL::DOUBLE AS ask FROM read_parquet('/path/to/file.parquet')",
  "partition": { "ticker": "SPX", "date": "2026-04-15" },
}
```

In practice, just tell Claude "import this file into `market.spot`" and it runs the sniff → map → import workflow for you. Any file DuckDB can read via `read_parquet`, `read_csv`, or `read_json` is a valid source — Parquet from a third-party vendor, a CSV you exported from another tool, whatever. The old `import_flat_file { date, asset_class, underlying, provider }` signature is gone; update any scripts or prompt templates that called it.

## Ticker Registry for OCC Roots

OCC option tickers use a "root" prefix that's often different from the canonical underlying. `SPXW250106C05000000` has root `SPXW` but belongs to the `SPX` underlying. `NDXP` is a P.M.-settled NDX weekly. Without a registry, every place in the system that groups contracts by underlying would either have to hardcode these mappings or guess — and guessing breaks `QuoteStore.readQuotes`, which requires every OCC ticker in a batch to resolve to the same underlying.

3.0 adds a registry that centralizes the mapping. Two new MCP tools let you inspect and extend it:

- `list_underlyings` — return every entry (bundled defaults + your additions) with a `source` field of `default`, `user`, or `user-override`.
- `register_underlying { underlying, roots }` — add a new mapping or override a bundled default. For example: `{"underlying": "BKNG", "roots": ["BKNG"]}` to support options on Booking Holdings, or `{"underlying": "SPX", "roots": ["SPX", "SPXW", "SPXQ", "SPXPM"]}` to add SPXPM alongside the bundled SPX roots.

### What ships bundled

The server ships with a small set of defaults for the most common index options, so quote resolution works out of the box for the primary use case without any configuration:

| Underlying | Roots                 | Why it's bundled                                                                                                                                          |
| ---------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `SPX`      | `SPX`, `SPXW`, `SPXQ` | Primary options underlying. Mixing monthlies + dailies in one batch is the common case — without this default, `QuoteStore.readQuotes` would reject them. |
| `VIX`      | `VIX`                 | Required for `compute_vix_context`.                                                                                                                       |
| `VIX9D`    | `VIX9D`               | Required for `compute_vix_context` (term-structure fields).                                                                                               |
| `VIX3M`    | `VIX3M`               | Required for `compute_vix_context` (term-structure fields).                                                                                               |
| `QQQ`      | `QQQ`, `QQQX`         | Nasdaq-100 ETF + index options root.                                                                                                                      |
| `NDX`      | `NDX`, `NDXP`         | Nasdaq-100 cash-settled index options (A.M. + P.M.).                                                                                                      |
| `RUT`      | `RUT`, `RUTW`         | Russell 2000 index options (monthly + weekly).                                                                                                            |
| `ES`       | `ES`                  | E-mini S&P 500 futures root (for users who ingest futures option data).                                                                                   |

If you don't trade any of these, they just sit in the registry unused — they're not populated with data, just mappings.

### How persistence works

Defaults live in code (`defaults.json`, compiled into the server binary) and are merged into the in-memory registry at every startup. They never touch disk on your side — you can't accidentally delete or corrupt them.

Your own additions are stored at `~/tradeblocks-data/market/underlyings.json` as a single JSON file. Adding a new underlying via `register_underlying` creates a `user` entry; registering an underlying that has a bundled default creates a `user-override` entry that replaces the default's root list for that underlying. Calling `unregister_underlying` on a user override reverts it to the bundled default; calling it on a pure user entry removes it entirely; bundled defaults themselves cannot be unregistered (but you can always override them).

This keeps your overrides portable and inspectable — a git-friendly JSON file that describes exactly what you've added on top of the shipped behavior. The file is empty on first start; everything you see in `list_underlyings` at that point is coming from the bundled defaults.

## SQL Can Introspect Parquet Files Directly

`run_sql` now accepts `read_parquet`, `read_csv`, and `read_json` calls, but only when every path argument resolves under `--data-root`. That means debug queries can introspect the canonical Parquet partitions:

```sql
SELECT COUNT(*) FROM read_parquet(
  '~/tradeblocks-data/market/spot/ticker=SPX/**/*.parquet',
  hive_partitioning=true
)
```

Paths outside the data root are rejected with a clear error. Hard blocks on `COPY TO`, `EXPORT`, `ATTACH`, `DETACH`, `read_text`, `write_csv`, and standalone `SET` are unchanged.

## Migrating from 2.x

If you're upgrading from 2.x:

- **Queries against `market.daily`, `market.intraday`, `market.date_context` will fail.** Use `market.spot_daily`, `market.spot`, and `market.enriched_context` instead. The server drops the old physical tables on open so stale references surface immediately rather than returning wrong data.
- **`import_flat_file` callers must pass `{file_path, dataset_type, select_sql, partition}`.** If you were calling with `{date, asset_class, underlying}`, rewrite using the `run_sql`-based sniff workflow described above.
- **Set `TRADEBLOCKS_PARQUET=true`** if you want Parquet-backed writes. Without it, 3.0 behaves like 2.x in terms of on-disk format (DuckDB tables) — but the new tools work either way.
- **Metadata migrations are Parquet-mode-only.** If you set `TRADEBLOCKS_PARQUET=true`, the first time the server opens a 2.x database it copies strategy profiles, block sync metadata, market import metadata, and flat-import log entries from their DuckDB tables to JSON files under `blocks/` and `market-meta/`. The DuckDB tables are left as-is (not dropped). If you stay on the DuckDB-mode default, nothing migrates and nothing changes — profile CRUD and the other tools continue to read and write the 2.x tables exactly as before.
- **Data root layout**: `~/tradeblocks-data/` is now organized as `database/` (local DuckDB files), `market/` (Parquet datasets + `underlyings.json` when `TRADEBLOCKS_PARQUET=true`), `market-meta/` (operational metadata JSON), `blocks/` (trades + profiles). If you do copy or sync the data root between machines, keep `database/` local per machine — DuckDB files shouldn't be shared.

# Design Changes

This section is for contributors, power users, and anyone maintaining code that extends TradeBlocks. Skip if you just care about using the features above.

## Store Abstraction Layer

Market data writes in 2.x went through a mix of direct `INSERT OR REPLACE INTO market.X` calls, `COPY TO` invocations in the Parquet path, and per-dataset helpers. In 3.0, all writes go through a store abstraction: `SpotStore`, `QuoteStore`, `ChainStore`, and `EnrichedStore` are abstract classes with two concrete implementations each — one that writes Parquet partitions via `COPY TO`, one that writes DuckDB rows via `INSERT OR REPLACE`. The factory `createMarketStores({parquetMode})` selects per-backend concretes once per process; every method on the store is monomorphic (no runtime mode checks).

Three write primitives exist on each store:

```ts
// Typed-row writes (from provider fetch)
spot.writeBars(ticker, date, bars);

// SELECT-driven writes (new in 3.0 — powers import_flat_file)
spot.writeFromSelect({ ticker, date }, selectSql);

// Coverage and reads
spot.readBars(ticker, from, to);
spot.readDailyBars(ticker, from, to);
spot.getCoverage(ticker, from, to);
```

All tool handlers, the ingestor, and the flat-file importer go through this layer — no tool reaches into raw SQL or raw Parquet paths anymore. If you extend TradeBlocks with a new market data source, implement it against these store methods and you get both backends for free.

## Provider Interface Expansion

The `MarketDataProvider` interface grew substantially between 2.3 and 3.0. In 2.3, a provider implemented bar fetching, snapshot fetching, and an optional quote fetch. In 3.0, it also declares its capabilities, participates in contract-list reference lookups, streams bulk quotes, and downloads flat files.

The new methods (all optional except `capabilities()`, `fetchBars`, and `fetchOptionSnapshot`):

```ts
interface MarketDataProvider {
  capabilities(): ProviderCapabilities    // declarative feature matrix
  fetchBars(...)                           // existing
  fetchOptionSnapshot(...)                 // existing
  fetchQuotes?(...)                        // existing, per-ticker NBBO
  fetchContractList?(...)                  // new: reference endpoint for contract metadata
  fetchBulkQuotes?(...)                    // new: async iterable, wildcard-by-underlying
  downloadFlatFile?(...)                   // new: raw file download
  downloadBulkData?(...)                   // new: download + filter + write Parquet
}
```

`ProviderCapabilities` is the linchpin. Instead of hardcoding `if (provider === "massive") ...` branches, tools check `provider.capabilities()` and dispatch accordingly:

```ts
interface ProviderCapabilities {
  tradeBars: boolean; // minute OHLC from trade aggregates
  quotes: boolean; // NBBO bid/ask
  greeks: boolean; // provider-computed greeks
  flatFiles: boolean; // bulk S3/file download
  bulkByRoot: boolean; // one call returns all strikes (ThetaData pattern)
  perTicker: boolean; // one call per OCC ticker (Massive/Polygon pattern)
  minuteBars: boolean;
  dailyBars: boolean;
  dataAvailability?: {
    // earliest available date per asset class
    option?: { from: string };
    index?: { from: string };
    stock?: { from: string };
  };
}
```

A tool that needs wildcard-by-underlying quotes (e.g., `fetch_quotes { underlyings: [...] }`) checks `caps.bulkByRoot` — on providers that don't support it, the tool returns `status: "unsupported"` with a clear error, rather than silently falling back or erroring deep in the fetch path.

**Adding your own provider**: Implementing the `MarketDataProvider` interface is the whole integration. Declare your capabilities, implement the methods your capabilities advertise, and the full TradeBlocks tool surface — `fetch_bars`, `fetch_quotes`, `fetch_chain`, `refresh_market_data`, enrichment, replay, greeks attribution — works against your provider automatically. Unsupported capabilities just return `unsupported`; the tools don't need to know anything about your provider specifically.

## LLM-as-Parser: the import_flat_file Design

The old `import_flat_file` hardcoded Massive-specific dispatch: `{date, asset_class, underlying, provider}` → S3 path → CSV parse → Parquet write. Every new source format or dataset type needed code changes to a format registry inside the server.

The new design pushes that intelligence to the LLM. Claude has `describe_database` (knows target schemas), `run_sql` with path-gated `read_parquet`/`read_csv` (can sniff any file), and the context to compose a bridging SELECT. The tool just accepts `{file_path, dataset_type, select_sql, partition}` and dispatches to the right store's `writeFromSelect`. Mode-routing is handled inside the store:

- **Parquet mode**: `COPY (select) TO spot/ticker=X/date=Y/data.parquet`
- **DuckDB mode**: `INSERT OR REPLACE INTO market.spot (...) <select>`

The tool itself contains zero provider code, zero format detection, and zero hardcoded parsers. Adding support for a new source format requires no server changes — Claude writes a SELECT and the ingest happens. If a new dataset type is needed (e.g., option trade prints), add a new store and a `writeFromSelect` impl; no dispatch-registry changes.

This principle — **the LLM is the intelligence layer** — applies more broadly. Tool signatures should accept typed configs from the caller rather than bake in per-provider or per-format decisions. It's documented in `CLAUDE.md` as a design guideline for new tools.

A lighter validator (`validateImportSelect`) gates the submitted SELECT: it still blocks `COPY TO`, `EXPORT`, `ATTACH`, `DETACH`, `read_text`, `write_csv`, and standalone `SET`, but allows `read_parquet`/`read_csv` from any path — the output is already sandboxed because the store composes the write path from `--data-root` + whitelisted partition values.

## JSON Metadata Stores

Metadata stores that used to live in DuckDB tables are now JSON files written atomically (write-then-rename). This removes a whole class of "DuckDB locked" errors when writing from multiple contexts, and makes per-block metadata portable without running the database engine:

- Strategy profiles — `{blocks_dir}/{block_id}/profiles/{strategy_name}.json`
- Per-block sync state — `{blocks_dir}/{block_id}/.sync-meta.json`
- Market sync metadata — `{data_root}/market-meta/sync-metadata.json` (per-source watermarks, content hashes)
- Flat-import log — `{data_root}/market-meta/flat-import-log.json` (dedupe ledger for flat-file ingests)
- Enrichment watermarks — `{data_root}/market-meta/enrichment-watermarks.json` (per-ticker last-computed date)

The profile CRUD tools (`list_profiles`, `get_strategy_profile`, `profile_strategy`, `delete_profile`) route by mode: in Parquet mode they read and write JSON files; in DuckDB mode they continue to read and write the 2.x `profiles.strategy_profiles` table. The switch is transparent to callers. When you first run with `TRADEBLOCKS_PARQUET=true`, a one-time migration copies existing profiles, sync metadata, market import metadata, and flat-import log entries from the DuckDB tables into JSON files. Each sub-migration is idempotent — if JSON files for a store already exist, it's skipped. The DuckDB tables are left intact (not dropped or modified), so flipping `TRADEBLOCKS_PARQUET=false` after migrating still reads the original 2.x data.
