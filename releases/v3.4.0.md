# TradeBlocks MCP Server 3.4.0 — Content-addressed market-data provenance

This release adds a public foundation for exact-byte provenance on canonical market-data Parquet writes.

## New public API

The `tradeblocks-mcp/market/provenance` export provides:

- versioned canonical JSON encoding with NFC normalization, integer-only numeric semantics, and exact-byte SHA-256 content addresses;
- a crash-safe, immutable, no-replace canonical JSON object store that addresses the exact encoded bytes once;
- versioned partition commit receipts with schema revision, logical coverage, quality counts, and append/repair classification;
- a market-root-owned receipt store with transactional file installation, append-only commit-event authority, rebuildable partition heads, generation-safe dead-owner recovery, verification, and orphan detection; and
- attempt-scoped receipt capture that leaves existing market store method contracts unchanged.

Bounded canonical market-dataset writers now supply stable dataset, partition, and relative-path identities to the provenance layer. Counts and logical coverage are read back from the completed Parquet bytes before installation. Exact-byte hashing and receipt persistence remain opt-in: they run only inside a partition commit attempt. Existing callers outside an attempt retain their prior write behavior and avoid the additional hashing work. Legacy whole-history enriched files are explicitly refused inside an attempt until they have bounded date-partitioned representations.

Partition installation now pins one unshared regular-file inode from verification through atomic rename and final authority checks. Symbolic links, multiply-linked files, non-regular files, symlinked target directories, and target entries that change during publication or inspection are rejected.

If a Parquet file is atomically published but the commit event or derived head cannot be completed, the write fails with an explicit orphan error while leaving the installed file available for authority-directed inspection and reconciliation.

## Upgrade and compatibility notes

- Existing 3.3.x `enriched/ticker=X/data.parquet` and
  `enriched/context/data.parquet` files remain readable through an immutable
  whole-file fallback; bounded slices win date-by-date when layouts coexist.
  The next enrichment/write lifecycle migrates supported XNYS sessions into
  bounded slices before consulting the enrichment watermark, so an advanced
  watermark cannot hide history. Existing slices are adopted rather than
  replaced during an active provenance attempt, making mixed-layout migration
  idempotent and authority-bearing. Legacy context rows missing any required
  completeness field are retained in the old file but intentionally omitted
  from canonical `enriched_context` slices and from the fallback join; joined
  reads therefore return null context for those dates until a complete context
  refresh is available.
- Parquet store bundles and their store instances are frozen, with immutable
  own method slots. Consumers that wrapped, monkey-patched, or spied directly
  on store instances must wrap the bundle before construction or use an
  external adapter. The public ticker-registry surface is a frozen read-only
  facade, but its lookups remain live so same-process register/unregister
  operations affect subsequent store reads.
- Canonical spot reads return `[]` only for true canonical absence. If only a
  legacy/global spot layout or excluded disk partitions are present, they now
  throw `MarketDataAuthorityError` instead of silently reporting no data.
- String-key risk-free-rate resolvers and their numeric wrappers now throw for
  malformed date keys or table values that are not exact to one basis point;
  the numeric wrappers otherwise retain their historical edge clamping.
  Canonical rate-slice publication additionally throws for stale requests
  after the bundled tail and lookahead-shaped requests before the bundled
  start.
- Bundled SOFR and 3-month Treasury inputs are refreshed through 2026-07-20.
  Canonical cutoff creation remains limited by both the bounded 2022-2030 XNYS
  calendar revision and the last bundled observation for each rate series; a
  later session fails closed until authoritative rate bytes are refreshed and
  the package is rebuilt.
