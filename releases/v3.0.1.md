# TradeBlocks MCP Server 3.0.1 — Symlink-aware entrypoint detection

A patch release fixing MCP server startup under Node version managers.

## Fix: entrypoint detection follows symlinks

The server decides whether to auto-start by checking whether it was run directly (rather than imported as a library). That check compared `process.argv[1]` against `import.meta.url` — but when the `tradeblocks-mcp` bin is invoked through a symlink (as nvm and other Node version managers do), `process.argv[1]` is the symlink path while `import.meta.url` is the resolved real path. The two never matched, so the server silently failed to start.

Startup now resolves both paths through `fs.realpath` before comparing, with a fallback to the plain resolved path if resolution fails. The server starts correctly whether it is invoked directly or through a version-manager symlink, and still won't auto-start when imported as a library.

Thanks to @concernedcat for the fix (#352).
